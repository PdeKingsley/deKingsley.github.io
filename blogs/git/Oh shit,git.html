    Oh Shit,git!?

    4/30/2022 git

    今日偶然发现了一个叫oh shit,Git!?的网站，里面的内容很有意思，让我们一一来看网站作者到底给我们分享了哪些小tip。

    1. 当我们在某一次提交之后导致一些东西被误删或者误加，亦或者是一次失败的merge污染了我们的本地仓库之后，我们有两条命令能够帮助我们逃离这样的窘境。

       git reflog
 git reset HEAD@{index}

    试一试吧，我们首先init一个仓库，创建一个readme文件并且提交，之后我们执行git log，我们会看到两次提交都会给我们显示出来。

    过了一会，我们又出于某不知名原因，不想要readme文件了。执行git reflog，可以看到如下的操作信息。

    我们执行git reset HEAD@{4},再执行git log,我们可以看到只剩下init这一条commit信息了。

    我们成功的让仓库回到了没有readme文件的状态，perfect！其实这里只是举个简单的例子，像如上这种需求，完全没必要这么做。但在复杂场景下，这么做可比一个文件一个文件的检查方便多了。

    1. 当我们的上一个提交需要一些变动时，或者是修改它的提交信息，又不想产生新的提交时，下面的两条命令可以帮助我们做到这一切。

       git commit --amend --no-edit
 git commit --amend

    没啥好说的，用就完了。

    1. 当我们将修改提交到错误的分支上时应该怎么补救呢？有两种可行方式

       #第一种
 git reset HEAD~ --soft
 git stash
 git checkout right_branch
 git add .
 git commit -m
 #第二种
 git checkout right_branch
 git cherry-pick master
 git checkout master
 git reset HEAD~ --hard

    两种方式都有效，选择哪个可自行判断。

    1. git diff显示的是未放入暂存区的文件的差异,git diff --staged才是放入暂存区的文件的差异。

    2. 当想要将之前隔了几个提交的那个提交所作的改动还原，可以通过git log得到那次提交的hashcode，之后执行git revert [hashcode]就ok了。

    3. 如果我们只想还原某个文件的变动，执行git checkout [hashcode] -- filepath即可。
